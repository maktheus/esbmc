\chapter{Metodologia}
\label{cap:metodologia}

Este capítulo descreve os procedimentos metodológicos adotados para verificação formal de sistemas de Inteligência Artificial Generativa, GenAI utilizando o verificador de modelos ESBMC (\textit{Efficient SMT-Based Context-Bounded Model Checker}). A investigação foi estruturada em seis estudos de caso complementares. O Caso~1 \'{e} organizado em uma hierarquia de quatro n\'{i}veis de verificabilidade, do neur\^{o}nio isolado ao bloco MLP SwiGLU estilo DeepSeek, culminando na arquitetura de abstra\c{c}\~{a}o Python sobre o \textit{core} nativo C++.

\section{Visão Geral da Abordagem}

A metodologia geral segue o ciclo de verificação formal baseado em SMT, conforme ilustrado na \autoref{fig:pipeline_esbmc}. O ESBMC opera convertendo o código-fonte C/C++ (ou \textit{stubs} ONNX traduzidos) para uma representação intermediária (\textit{GOTO-Programs}), realizando execução simbólica e codificando as restrições em fórmulas SMT (\textit{Satisfiability Modulo Theories}). O \textit{solver} SMT, Z3 ou Bitwuzla então determina se existe uma atribuição de valores de entrada capaz de violar as propriedades especificadas   produzindo um contra-exemplo explícito em caso afirmativo.

    \end{center}

Para cada caso de uso, foram definidas:
\begin{alineas}
    \item \textbf{Pré-condições}: restrições sobre os valores de entrada, ex.: intervalo de valores, tamanho de buffers;
    \item \textbf{Propriedades de Segurança}: asserções formais que o sistema deve satisfazer para todo valor de entrada válido;
    \item \textbf{Métricas de Avaliação}: tempo de verificação, escalabilidade e eficácia na detecção de falhas.
\end{alineas}

\section{Caso 1 (N\'ivel 1) Verifica\c{c}\~ao Formal de Neur\^onio com Ativa\c{c}\~ao ReLU}

O primeiro caso de uso corresponde ao \textbf{N\'ivel 1} da hierarquia de verifica\c{c}\~ao proposta: a verifica\c{c}\~ao de propriedades matem\'aticas de unidades computacionais elementares de redes neurais, implementadas em Python com anota\c{c}\~oes de tipo estritas. O objetivo central \'e demonstrar que o ESBMC \'e capaz de \textbf{provar formalmente}, para \textit{qualquer} entrada dentro de um dom\'inio especificado, que a sa\'ida de um neur\^onio artificial com ativa\c{c}\~ao ReLU sempre obedece a limites te\'oricos calculados analiticamente.

\subsection{Arquitetura do Modelo}

O modelo implementado (\texttt{neuron.py}) consiste em um \'unico neur\^onio artificial com dois pesos fixos, um \textit{bias} e fun\c{c}\~ao de ativa\c{c}\~ao ReLU:

\begin{equation}\label{eq:neuron}
out = \text{ReLU}(w_1 \cdot x_1 + w_2 \cdot x_2 + b)
\end{equation}

onde $w_1 = 0.5$, $w_2 = -0.2$, $b = 0.1$, e $\text{ReLU}(z) = \max, 0, z$.

O Python puro foi escolhido sem depend\^encias de \textit{frameworks} din\^amicos como NumPy ou PyTorch pois o \textit{frontend} do ESBMC requer anota\c{c}\~oes de tipo expl\'icitas (\texttt{float}) para traduzir corretamente o c\'odigo para a representa\c{c}\~ao intermedi\'aria \texttt{GOTO-Programs}.

\begin{lstlisting}[language=Python, numbers=none]
def relu, x: float -> float:
    if x > 0:
        return x
    return 0.0

def neuron, x1: float, x2: float -> float:
    w1: float = 0.5
    w2: float = -0.2
    bias: float = 0.1
    activation: float = , x1 * w1 + , x2 * w2 + bias
    return relu, activation

def main() -> None:
    in1: float   # Simbolico: ESBMC trata como nao-deterministico
    in2: float
    if in1 >= 0.0 and in1 <= 1.0 and in2 >= 0.0 and in2 <= 1.0:
        out: float = neuron, in1, in2
        assert out >= 0.0   # Prop1: ReLU garante nao-negatividade
        assert out <= 0.6   # Prop2: bound superior calculavel
main()
\end{lstlisting}

\subsection{Propriedades de Verifica\c{c}\~ao e Prova Anal\'itica}

A propriedade de seguran\c{c}a pode ser verificada analiticamente, servindo de \textit{ground truth} para validar o resultado do \textit{solver} SMT:

\begin{alineas}
    \item \textbf{Pr\'e-condi\c{c}\~ao} (regi\~ao de hip\'otese): $x_1, x_2 \in [0.0, 1.0]$;
    \item \textbf{Limite Superior}: A sa\'ida m\'axima ocorre em $x_1 = 1.0$, $x_2 = 0.0$ (pois $w_2 < 0$):
    \[
    out_{\max} = \text{ReLU}(0.5 \cdot 1.0 + , -0.2 \cdot 0.0 + 0.1) = \text{ReLU}(0.6) = 0.6;
    \]
    \item \textbf{Limite Inferior}: Como ReLU $\geq 0$ por defini\c{c}\~ao, $out_{\min} = 0.0$;
    \item \textbf{P\'os-condi\c{c}\~ao} (\textit{assert}): $0.0 \leq out \leq 0.6$, $\forall, x_1, x_2 \in [0,1]^2$.
\end{alineas}

O verificador formal foi executado com os argumentos:

\begin{alineas}
    \item \texttt{-{}-floatbv}: codifica\c{c}\~ao precisa de aritm\'etica de ponto flutuante via \textit{Bitvectores} IEEE~754;
    \item \texttt{-{}-k-induction}: estrat\'egia de prova por indu\c{c}\~ao-$k$, percorrendo todos os caminhos simb\'olicos do programa $k$ vezes, buscando uma atribui\c{c}\~ao que viole as asser\c{c}\~oes.
\end{alineas}

\section{Caso 2   Verificação do Motor de Inferência, Kernels C++}

O segundo caso focou na verificação de segurança de memória em kernels C++ análogos aos utilizados em motores de inferência como \texttt{llama.cpp}. Foi implementado um kernel simplificado de atenção \textit{dot-product} (\texttt{attention\_kernel.cpp}) com alocação dinâmica de memória e laços de acesso a arrays.

A abordagem adotada emprega entradas simbólicas não-determinísticas ($\texttt{seq\_len} \in [1, 10]$) e verifica:
\begin{alineas}
    \item Ausência de \textit{buffer overflows} nos acessos \texttt{query[i]} e \texttt{key[i]};
    \item Ausência de vazamentos de memória (\textit{memory leaks}).
\end{alineas}

Adicionalmente, um kernel de multiplicação de matrizes (\textit{GEMM}) com \textit{tiling} foi implementado (\texttt{matmul\_kernel.cpp}) para análise de escalabilidade, variando a dimensão $N \in \{2, 3, 4, 5, 6\}$.

O comando utilizado foi o \texttt{esbmc} com os parâmetros: \\
\texttt{attention\_kernel.cpp -{}-multi-property -{}-memory-leak-check -{}-overflow-check}.

\section{Caso 3 --- Loop Neuro-Simb\'{o}lico, Agente LLM + ESBMC}

O terceiro caso simula um ciclo aut\^{o}nomo de gera\c{c}\~{a}o e verifica\c{c}\~{a}o de c\'{o}digo, modelando o fluxo \textbf{Ralph Loop}: o agente requisita ao LLM um trecho de c\'{o}digo C, submete ao ESBMC e, caso o verificador produza um contra-exemplo, repassa o diagn\'{o}stico ao LLM para refinamento imediato. O script principal \'{e} \texttt{mock\_agent.py}.

\subsection{Arquitetura do Agente}

O agente executa at\'{e} 5 itera\c{c}\~{o}es. Cada itera\c{c}\~{a}o \'{e} composta por tr\^{e}s etapas:

\begin{alineas}
    \item \textbf{Gera\c{c}\~{a}o LLM}: simula atraso de 0,5 a 2,0 segundos e retorna um trecho de c\'{o}digo C. Na itera\c{c}\~{a}o 0, o c\'{o}digo cont\'{e}m deliberadamente um \textit{buffer overflow} via \texttt{strcpy} num buffer de 10 bytes com entrada de 20 bytes; a partir da itera\c{c}\~{a}o 1, o c\'{o}digo \'{e} corrigido com \texttt{strncpy};
    \item \textbf{Verifica\c{c}\~{a}o ESBMC}: o verificador \'{e} invocado com \texttt{--overflow-check --memory-leak-check}. A sa\'{i}da \'{e} analisada em busca de \texttt{VERIFICATION SUCCESSFUL};
    \item \textbf{Feedback sem\^{a}ntico}: em caso de falha, o contra-exemplo do ESBMC, contendo o \textit{offset} exato da viola\c{c}\~{a}o e o estado do programa naquele ponto, \'{e} repassado como prompt de corre\c{c}\~{a}o ao LLM.
\end{alineas}

Os tempos e resultados de cada itera\c{c}\~{a}o s\~{a}o registrados em \texttt{case3\_agent\_stats.csv}:

\begin{lstlisting}[language=Python, numbers=none]
cmd = ["esbmc", filename,
       "--overflow-check",
       "--memory-leak-check",
       "--no-pointer-check",
       "--smtlib"]
result = subprocess.run(cmd, capture_output=True, text=True)
success = "VERIFICATION SUCCESSFUL" in result.stdout
\end{lstlisting}

\subsection{Propriedade Verificada}

O ESBMC detecta que \texttt{strcpy(buffer, input)} com \texttt{sizeof(buffer)=10} e \texttt{sizeof(input)=20} constitui um acesso fora dos limites da mem\'{o}ria, produzindo um contra-exemplo concreto com o \textit{offset} exato da viola\c{c}\~{a}o que \'{e} devolvido como feedback ao agente LLM.


\section{Caso 4 --- Controladores Neurais e Planta de Sistema em Malha Fechada}

O quarto caso verifica formalmente a seguran\c{c}a de um controlador PID digital operando em malha fechada sobre uma planta t\'{e}rmica, sob condi\c{c}\~{o}es de ru\'{i}do n\~{a}o-determin\'{i}stico nos sensores. A implementa\c{c}\~{a}o (\texttt{pid\_controller.c}) mimetiza sistemas embarcados cr\'{i}ticos onde ru\'{i}dos de sensor comprometem a estabilidade.

\subsection{Modelo da Planta e do Controlador}

A planta t\'{e}rmica evolui conforme:

\begin{equation}\label{eq:plant}
T_{k+1} = \max\!\bigl(20,\; T_k + ,( u_k \cdot r_h - r_c ,) \cdot \Delta t\bigr)
\end{equation}

onde $r_h = 0,1\,^{\circ}\text{C/\%}$, $r_c = 2,0\,^{\circ}\text{C}$ e $\Delta t = 1$\,s. O controlador PID com $K_p = 1{,}0$, $K_i = 0{,}1$, $K_d = 0{,}5$ calcula:

\begin{equation}\label{eq:pid}
u_k = K_p\,e_k + K_i\!\sum_{j=0}^{k} e_j\,\Delta t + K_d\,\frac{e_k - e_{k-1}}{\Delta t}, \quad u_k \in [0, 100]
\end{equation}

\subsection{Inje\c{c}\~{a}o de Caos e Propriedade de Seguran\c{c}a}

O sensor recebe ru\'{i}do n\~{a}o-determin\'{i}stico escolhido pelo pr\'{o}prio \textit{solver}:

\begin{lstlisting}[language=C, numbers=none]
float noise = nondet_float();
__ESBMC_assume(noise >= -5.0f && noise <= 5.0f);
float measured = true_temp + noise; // Pior caso buscado pelo solver
assert(temp < 150.0f);             // Invariante de seguranca
\end{lstlisting}

A invariante de seguran\c{c}a verificada para $\forall k \in [0, 10]$ e $\forall\,\text{noise} \in [-5, +5]$ \'{e}:

\begin{equation}\label{eq:safety}
T_k < T_{\text{MAX}} = 150\,^{\circ}\text{C}
\end{equation}

O ESBMC executa \textit{Bounded Model Checking} com \texttt{--unwind~11 --floatbv}, desenrolando simbolicamente todos os 10 passos do la\c{c}o e explorando exaustivamente o espa\c{c}o de ru\'{i}do para tentar violar a asser\c{c}\~{a}o.


\section{A Dicotomia Arquitetural: Orquestracao Python vs. Core Nativo C/C++}

Embora as provas matemáticas rigorosas recaiam intrinsecamente sobre o \textit{solver} associado ao verificador, capaz de destrinchar representações de memória estritas originadas do C/C++, a arquitetura global adota linguagens de alto nível (\textit{scripts} em Python) como sua principal "Mente Arquitetural". Por limitações de tipagem dinâmica e natureza interpretada, é inviável modelar restrições temporais SMT diretamente sobre o código \texttt{.py}. Em vez disso, a metodologia opera numa simbiose de responsabilidades:

\begin{alineas}
    \item \textbf{C/C++ , O Motor Matemático:} Abriga as implementações nucleares sob teste   plantas de malhas fechadas dinâmicas, camadas preditivas limitadas (\textit{Dense, Relu, Attention}) e suas asserções lógicas (\texttt{\_\_ESBMC\_assert}).
    \item \textbf{Python, O Orquestrador de Testes e Caos:} Age encapsulando o verificador (\texttt{core\_verify/}). Esta camada dinamicamente formata perturbações estatísticas não-lineares, injecao de caos sistemico, transcrevendo-as dentro do esqueleto dos \textit{Stubs} em C, e automatiza as rotinas de estresse varrendo dezenas de operacoes matriciais em iteracoes de Integracao Continua.
\end{alineas}

\section{Caso 5 e 6   Ralph Loop Architect e RL Actor-Critic}

Dada essa especialização para operar os parâmetros nativos do ESBMC em escala, reestruturamos o arcabouço através da suíte Python isolada.

No \textbf{Caso 5}, reestruturamos o \textit{Mock LLM Agent} através de orquestradores desacoplados capazes de extrair as métricas de quebra SMT (\textit{Array Bounds}, \textit{Buffer Overflows}) através de Expressoes Regulares direcionais (\textit{Regex Trace Parser}). Essa instrumentação gera a prova metodológica do isolamento de falhas \textit{vis-à-vis} a visão contínua neural vs a representação disjunta SMT. A Figura abaixo, via listagem de código denota a classe \texttt{FeedbackTrace}, responsável pelo mapeamento das saídas brutas do verificador em categorias semânticas, como vazamento de memória e índices fora de limite:

\begin{lstlisting}[language=Python, numbers=none]
class FeedbackTrace:
    # Mapeamento de rastreios ESBMC nativos para excecoes categorizadas
    PATTERNS = {
        "array out of bounds": r"(?i)array bounds violated",
        "arithmetic overflow": r"(?i)arithmetic overflow",
        "memory leak": r"(?i)memory leak|dereference failure",
        "assertion failure": r"(?i)assertion.*failed|violated property",
    }
    
    def parse_violations, self:
        if self.is_timeout:
            self.violations.append, "target timeout"
            return
        for name, pattern in self.PATTERNS.items():
            if re.search, pattern, self.raw_output:
                self.violations.append, name
\end{lstlisting}

No \textbf{Caso 6}, estressamos o arcabouço através da simulação física formal de Restrições em Aprendizado por Reforço (\textit{Actor-Critic RL}). Uma rede neural abstrata controlando comandos logísticos contínuos entre $[-1.0, 1.0]$ foi transcrita em C e submetida à exploração não-linear profunda com o argumento \texttt{-{}-floatbv}, objetivando caçar estados matemáticos periféricos aos limites seguros operacionais do mecanismo em tempo de implantação robótica (\textit{Safety Shield}).

\section{Fluxo de Desenvolvimento e Integração Contínua}

Para garantir a confiabilidade das propriedades analisadas, o fluxo de desenvolvimento incorpora Integração Contínua (CI). A cada submissão, automações (via \textit{GitHub Actions}) provisionam as versões canônicas do ESBMC, resolvendo \textit{pipelines} de testes que cruzam \textit{scripts} em Python e os programas fontes da verificação:

\begin{lstlisting}[language=bash, numbers=none]
jobs:
  formal-verification:
    runs-on: ubuntu-latest
    steps:
    - name: Install ESBMC Binary
      run: |
        wget https://github.../v8.0.0/esbmc-linux.zip
        unzip esbmc-linux.zip -d bin/
        export PATH=$PATH:$(pwd)/bin/
        
    - name: Run Pytest Formal Verification
      run: |
        cd pibic
        pytest tests/ -v
\end{lstlisting}

Internamente, grande parte das avaliações do controlador neural lidou-se localmente providenciando as execuções de compilação simbólica baseadas em comandos chave do ESBMC. A validação de precisão matemática exige pragmas restritivas, por exemplo, determinando o tamanho das palavras flutuantes e verificações robustas:
\begin{alineas}
    \item \texttt{esbmc pid\_template.c -{}-floatbv -{}-k-induction}: Realiza a prova de malha de controle através de análise de indução-$k$, crucial para atestar a convergência segura frente aos laços da temporalidade dos modelos interativos.
    \item \texttt{esbmc attention\_kernel.cpp -{}-multi-property -{}-memory-leak-check}: Aborda violações ortogonais de integridade em código C++, identificando desvios na alocação da pilha (\textit{stack}) em tensores das arquiteturas \textit{Transformers}.
\end{alineas}
