Este capítulo apresenta os resultados parciais obtidos, organizados por caso de uso. Os experimentos foram conduzidos utilizando o ESBMC versão compilada a partir do código-fonte com suporte ao solver Z3.

\section{Caso 1 Verifica\c{c}\~ao Formal de Neur\^onio com ReLU}

\subsection{Metodologia de Execu\c{c}\~ao}

A verifica\c{c}\~ao do neurônio (\texttt{neuron.py}) foi executada localmente via linha de comando:

\begin{lstlisting}[language=bash, numbers=none]
esbmc pibic/1_python_models/neuron.py --floatbv --k-induction
\end{lstlisting}

O \textit{frontend} Python do ESBMC (baseado em \texttt{ast2json}) converte o c\'odigo-fonte diretamente para \texttt{GOTO-Programs}, a representa\c{c}\~ao intermedi\'aria interna do verificador. A execu\c{c}\~ao simb\'olica trata as vari\'aveis n\~ao inicializadas (\texttt{in1}, \texttt{in2}) como valores n\~ao-determin\'isticos, e a estrutura condicional \texttt{if in1 >= 0.0 and ...} \'e automaticamente convertida em uma \textit{pre-condi\c{c}\~ao} (\texttt{\_\_ESBMC\_assume}) pelo \textit{solver}.

\subsection{Resultado da Verifica\c{c}\~ao}

O ESBMC retornou o veredicto \textbf{VERIFICATION SUCCESSFUL} para ambas as asser\c{c}\~oes:

\begin{alineas}
    \item \textbf{Propriedade 1} (\texttt{out >= 0.0}): provada por indu\c{c}\~ao-$k$. O operador ReLU garante n\~ao-negatividade trivialmente: qualquer combina\c{c}\~ao linear que resulte em valor negativo \'e truncada a zero anteriormente;
    \item \textbf{Propriedade 2} (\texttt{out <= 0.6}): provada pelo \textit{solver} Z3 via Bitvectores IEEE~754. O estado m\'aximo atingível \'e $out = 0.6$, confirmado tanto pela prova anal\'itica ($out_{\max} = \text{ReLU}(0.5 - 0.0 + 0.1) = 0.6$) quanto pelo resultado \textit{UNSAT} ou seja, n\~ao existe nenhuma atribui\c{c}\~ao de $(x_1, x_2) \in [0,1]^2$ que fa\c{c}a $out > 0.6$.
\end{alineas}

A \autoref{fig:case1_plot} ilustra, de forma visual, a superf\'icie de sa\'idas do neur\^onio sobre todo o dom\'inio de entrada $[0,1]^2$ e a distribui\c{c}\~ao dos valores de sa\'ida obtidos por amostragem densa ($10^4$ pontos), comprovando que 100\% das amostras respeitam os \textit{bounds} formalmente verificados.

\subsection{An\'alise dos Resultados e Discuss\~ao}

Os resultados do Caso 1 evidenciam tr\^es contribui\c{c}\~oes relevantes:

\begin{alineas}
    \item \textbf{Coer\^encia anal\'itica}: O \textit{solver} SMT, operando sobre bitvectores de ponto flutuante, reproduz com precis\~ao o c\'alculo anal\'itico. O limite $out_{\max} = 0.6$ n\~ao \'e uma estimativa: \'e uma \textbf{prova matem\'atica} exaustiva sobre todos os valores poss\'iveis do dom\'inio;

    \item \textbf{Viabilidade do Frontend Python}: Embora o \textit{frontend} Python do ESBMC exija que o c\'odigo seja escrito com tipagem est\'atica expl\'icita e evitando constru\c{c}\~oes din\^amicas (listas heterog\^eneas, chamadas a \textit{frameworks} externos), o experimento confirma que a abordagem \'e \textbf{vi\'avel para subconjuntos estritos} de Python, representando um caminho promissor para verificar componentes cr\'iticos de modelos reais sem reescrev\^e-los em C;

    \item \textbf{Escalabilidade e Limita\c{c}\~oes}: O tempo de verifica\c{c}\~ao do Caso 1 foi inferior a 5 segundos para o modelo de neur\^onio \'unico. Em contraste, a aplica\c{c}\~ao direta da mesma abordagem para redes maiores (como a MLP em \texttt{mlp.py} com duas camadas e 3 neur\^onios) j\'a exige reestrutura\c{c}\~ao do c\'odigo e aumenta o espa\c{c}o de busca do \textit{solver}. Isso indica que a verifica\c{c}\~ao de modelos Python escala com a complexidade da rede de forma muito mais r\'apida do que a verifica\c{c}\~ao de c\'odigo C equivalente.
\end{alineas}

Em suma, o Caso 1 serve como prova de conceito da integra\c{c}\~ao ESBMC-Python: o verificador, munido dos argumentos \texttt{-{}-floatbv -{}-k-induction}, \'e capaz de consumir c\'odigo Python bem tipado, construir o modelo SMT correspondente e emitir uma \textbf{garantia formal absolutamente confi\'avel} de que as propriedades an\'alises s\~ao mantidas para todo o espa\c{c}o de entrada.



\subsection{Extens\~{a}o ao Dom\'{i}nio Generativo: Verifica\c{c}\~{a}o Modular de Transformers}

O Caso~1 prova a viabilidade da abordagem para um neur\^{o}nio isolado. A quest\~{a}o central de pesquisa \'{e}: \textbf{como essa abordagem escala para arquiteturas generativas modernas} como DeepSeek, Llama ou GPT? A resposta consiste em uma hierarquia de verificabilidade, diretamente ancorada no QNNVerifier~\cite{gopinath2021verifying}:

\begin{alineas}
    \item \textbf{N\'{i}vel 1, atual}: Neur\^{o}nio \'{u}nico ou MLP pequena com pesos fixos. Solver conclui em segundos, propriedades provadas exaustivamente;

    \item \textbf{N\'{i}vel 2}: Redes de classifica\c{c}\~{a}o com quantiza\c{c}\~{a}o \texttt{Fixed16/Fixed32}, como as 52 topologias do QNNVerifier. Alto desempenho com otimiza\c{c}\~{o}es de \textit{slicing} e infer\^{e}ncia de intervalos;

    \item \textbf{N\'{i}vel 3}: Blocos isolados de Transformer, cabeca de aten\c{c}\~{a}o com dimens\~{a}o de embedding reduzida e pesos fixos quantizados. \textbf{Implementado neste trabalho} em \texttt{attention\_block\_stub.c}, demonstrado a seguir;

    \item \textbf{N\'{i}vel 4}: Modelos completos como DeepSeek-R1 ou Llama-3.8B. A verifica\c{c}\~{a}o direta de todo o modelo \'{e} computacionalmente invi\'{a}vel. A estrat\'{e}gia praticada \'{e} a \textbf{verifica\c{c}\~{a}o modular por componente}: os pesos de cada camada s\~{a}o fixados como constantes e as entradas s\~{a}o tratadas como vari\'{a}veis simb\'{o}licas constrangidas a uma vizinhan\c{c}a $\mathcal{B}_\epsilon(x_0)$.
\end{alineas}

\subsection{Caso de Verifica\c{c}\~{a}o Modular: Cabeca de Aten\c{c}\~{a}o de Transformer}

O arquivo \texttt{attention\_block\_stub.c} implementa uma cabeca de aten\c{c}\~{a}o \textit{single-head} com dimens\~{a}o $d=4$ e sequ\^{e}ncia $\text{SEQ}=2$, com pesos $W_Q, W_K, W_V \in \mathbb{R}^{4 \times 4}$ fixados como constantes, representando uma camada quantizada de LLM. As entradas $x_0, x_1 \in [-1,1]^4$ s\~{a}o vari\'{a}veis simb\'{o}licas n\~{a}o-determin\'{i}sticas.

S\~{a}o verificadas tr\^{e}s propriedades formais:

\begin{equation}
\text{P1:}\quad s_{ij} = \frac{q_i \cdot k_j}{\sqrt{d}} \in [-3.0,\, 3.0] \quad \forall\, x_0, x_1 \in [-1,1]^4
\end{equation}

\begin{equation}
\text{P2:}\quad \alpha_{ij} \in [0,\,1] \;\wedge\; \textstyle\sum_j \alpha_{ij} \in [0.99,\,1.01]
\end{equation}

\begin{equation}
\text{P3:}\quad \| \text{out}_i \|_{\infty} \leq 1.0 \quad \text{(norma L-}\infty\text{ da sa\'{i}da da cabeca)}
\end{equation}

O trecho central do \textit{stub} ilustra como o ESBMC recebe entradas simb\'{o}licas e verifica as propriedades via \texttt{assert}:

\begin{lstlisting}[language=C, numbers=none]
/* Entrada simbolica: token embeddings em [-1, 1]^D */
float x0[D], x1[D];
for (int i = 0; i < D; i++) {
    x0[i] = nondet_float();
    __ESBMC_assume(x0[i] >= -1.0f && x0[i] <= 1.0f);
}
/* Projeta Q, K, V; calcula scores e softmax */
float s00 = dot(q0, k0) * 0.5f;   /* 1/sqrt(4) */
assert(s00 >= -3.0f && s00 <= 3.0f);    /* P1 */
assert(a00 >= 0.0f && a00 <= 1.0f);     /* P2 */
assert(out0[i] >= -1.0f && out0[i] <= 1.0f); /* P3 */
\end{lstlisting}

O comando de verifica\c{c}\~{a}o \'{e}:

\begin{lstlisting}[language=bash, numbers=none]
esbmc verification/attention_block_stub.c \
      --floatbv --unwind 4 \
      --overflow-check --no-unwinding-assertions
\end{lstlisting}

Este experimento demonstra que a \textbf{verifica\c{c}\~{a}o modular de componentes de LLMs \'{e} vi\'{a}vel} com o ESBMC: fixando pesos e restringindo entradas simb\'{o}licas a uma vizinhan\c{c}a compacta, o \textit{solver} SMT prova exaustivamente propriedades de seguran\c{c}a num\'{e}rica para toda uma cabeca de aten\c{c}\~{a}o. A mesma metodologia pode ser escalada para verificar o \textit{logit clamp} final, a fun\c{c}\~{a}o de ativa\c{c}\~{a}o SiLU das camadas MLP do DeepSeek, ou quaisquer componentes cr\'{i}ticos de um LLM quantizado.
